ROUND 1 — investigate why find() returns null
- Traced exec_method_in_class_hierarchy for find/add.
- `add` IS dispatched (handle 5 = the domain schedule); m_successors
  goes 0->1 on first add, so the graph gets populated.
- `find` is NEVER dispatched to the uvm_phase method — no trace.
- Conclusion: `common.find(...)` is intercepted before reaching the
  class method, almost certainly by the SV array `.find` locator
  handling. find() never runs -> returns 0 -> COMP/INTERNAL.

ROUND 2 — find() intercepted as array locator
- eval_builtin_method treated ANY `.find()` as the SV array locator.
- Gated the find/find_* builtins on the receiver actually being an
  array/queue/assoc. find() now dispatches to the uvm_phase method.

ROUND 3 — find() recursion traced
- find walks the phase graph h5->h8->h12->...; reaches h8 (m_imp=7)
  while searching for imp 7. The phase==m_imp match should fire.
- find() still returned null despite reaching the matching node.

ROUND 4 — implicit function-return variable (THE FIX)
- uvm_phase::find sets its result via `find = ...` (the implicit
  return variable named after the function), with no explicit
  `return`. exec_method_in_class_hierarchy only consulted
  `return_value` -> returned 0.
- Fix: seed a `<fnname>` local; after the body, fall back to that
  local when no explicit return ran. General fix for all class
  functions using the implicit-return convention.
- RESULT: COMP/INTERNAL gone. uvm_test_top component is created;
  reports now show the component path (`.uvm_test_top`).

ROUND 5 — characterize new stop point + verify
- UVM run now gets PAST component creation: uvm_test_top is built,
  reports are component-tagged. New output:
    UVM_WARNING @ 0: .uvm_test_top [INVLCMDARGS]
  then "Simulation finished at time 0".
- Next blocker: phase EXECUTION. uvm_root::run_test does

    #0;
    wait (m_phase_all_done == 1);
  The forked phase runner does not actually run the phases, so no

SECOND 5-ROUND PASS — phase-execution engine
================================================================

ROUND 1 — does m_run_phases / execute_phase run?  NO.
- Traced exec_method_in_class_hierarchy for the phase-execution
  methods. None of m_run_phases / execute_phase / build_phase / 
  run_phase ever execute.

ROUND 2 — wait() is non-blocking in nested calls
- The StatementKind::Wait exec_statement arm only evaluates the
  condition once: true -> run body, false -> fall through. It does
  NOT suspend.
- A proper blocking Wait exists, but only at process top-level
  (run_process_stmts). `wait` reached through nested task/method
  calls (as in uvm_root::run_test) uses the non-blocking arm.

ROUND 3 — run_events_until never called
- `#delay` mid-task drains the event queue via run_events_until,
  which would run the forked m_run_phases. But the trace shows
  run_events_until is never invoked during the UVM run.
- The fork IS scheduled: "ParBlock JoinNone scheduled pids=[1] at t=0".

ROUND 4 — execution does NOT stop at the fork
- After the fork: break=false, finished=false, no return_value.
- So run_test keeps going past the fork.

ROUND 5 — run_test runs straight through, abandoning the phase runner
- run_test executes all ~22 body statements in sequence and returns;
  the forked m_run_phases (pid 1, queued at t=0) is never run.
- Neither `#0` (no Delay/run_events_until trace) nor
  `wait(m_phase_all_done==1)` yields control to the scheduler.

ROOT CAUSE (architectural)
- uvm_root::run_test relies on cooperative concurrency:
    fork m_run_phases join_none;  #0;  wait(m_phase_all_done==1);
  This needs process SUSPENSION from arbitrary nested-call depth —
  a `wait`/`#0` deep inside task/method calls must suspend the whole
  call stack and resume when the scheduler says so.
- xezim suspends a process only at its top-level statement list
  (run_process_stmts). Blocking constructs reached through nested
  calls fall through. This is the simulation concurrency kernel —
  a major change (coroutine/continuation model or a re-entrant
  scheduler that can unwind+restore the interpreter call stack).

- The previous pass's two fixes (implicit function-return variable,
